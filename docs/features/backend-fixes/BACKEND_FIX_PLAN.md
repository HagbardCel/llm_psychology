# Backend API Fix Plan - Revised

## Goal Description

Address the critical backend API issues identified in `BACKEND_API_VERIFICATION_RESULTS.md` to ensure the frontend can successfully create user profiles, create therapy plans, and retrieve therapy styles.

**Architecture Principle**: Business logic lives in the orchestration layer. HTTP endpoints are thin wrappers that delegate to orchestrator methods. This ensures consistency between HTTP and WebSocket interfaces and prevents code duplication.

## Critical Issues Identified

1. ❌ **POST /api/user/profile** returns 500 error
2. ❌ **POST /api/therapy/plan** returns "not implemented"
3. ⚠️ **GET /api/therapy/styles** returns empty array instead of dynamic list

**Problem:**

```python
async def create_therapy_plan(
    self, user_id: str, therapy_style: str
) -> TherapyPlan:
    """
    Create therapy plan with selected therapy style.

    This encapsulates business logic for plan creation,
    callable from both HTTP and WebSocket interfaces.

    Args:
        user_id: User identifier
        therapy_style: Selected therapy style ("freud", "jung", "cbt")

    Returns:
        Created TherapyPlan

    Raises:
        ValueError: If validation fails
    """
    try:
        # Validate therapy style
        from services.style_service import style_service
        if therapy_style not in style_service.get_available_styles():
            raise ValueError(f"Invalid therapy style: {therapy_style}")

        # Get user profile
        trio_db_service = self.service_container.get("trio_db_service")
        profile = await trio_db_service.get_user_profile(user_id)
        if not profile:
            raise ValueError(f"User profile not found: {user_id}")

        # Check if plan already exists
        existing_plan = await trio_db_service.get_latest_therapy_plan(user_id)
        if existing_plan and existing_plan.version == 1:
            logger.info(f"Therapy plan already exists for {user_id}, returning existing")
            return existing_plan

        # Create minimal therapy plan (version 1)
        # Detailed plan_details will be generated by PlanningAgent during first session
        plan = TherapyPlan(
            plan_id=str(uuid.uuid4()),
            user_id=user_id,
            created_at=datetime.now(),
            updated_at=datetime.now(),
            plan_details={
                "focus": "To be determined in first therapy session",
                "goals": [],
                "techniques": [],
                "themes": []
            },
            version=1,
            selected_therapy_style=therapy_style
        )

        # Save plan
        success = await trio_db_service.save_therapy_plan(plan)
        if not success:
            raise RuntimeError("Failed to save therapy plan to database")

        # Update user status to PLAN_COMPLETE
        profile.status = UserStatus.PLAN_COMPLETE
        profile.updated_at = datetime.now()
        await trio_db_service.save_user_profile(profile)

        logger.info(f"Created therapy plan for {user_id} with style {therapy_style}")
        return plan

    except Exception as e:
        logger.error(f"Error creating therapy plan: {e}", exc_info=True)
        raise
```

## Implementation Details

### Error Handling Strategy

As requested, the implementation follows a **fail-fast** approach:

1. **Expected Errors** (validation, not found, etc.):
   - Return 400/404 with error message
   - Log with `logger.error()`
   - Do NOT crash

2. **Unexpected Errors** (database failures, bugs, etc.):
   - Log with `logger.error(..., exc_info=True)` for full traceback
   - **Re-raise** to crash the request
   - System outputs full stack trace

### Data Storage

**Therapy Style Storage:**
- Stored in `therapy_plans.selected_therapy_style` column (existing schema)
- No schema changes required
- Version 1 plan created by HTTP endpoint with minimal `plan_details`
- Later versions (2, 3, 4...) created by PlanningAgent with full LLM-generated content

**User Status Flow:**
```
POST /api/user/profile
  → UserStatus.PROFILE_ONLY (new profile)

POST /api/therapy/plan
  → UserStatus.PLAN_COMPLETE (ready for therapy)
```

### Duplication Prevention

The orchestrator method checks for existing version-1 plans:
```python
existing_plan = await trio_db_service.get_latest_therapy_plan(user_id)
if existing_plan and existing_plan.version == 1:
    return existing_plan  # Don't create duplicate
```

This ensures only ONE version-1 plan per user, even if the endpoint is called multiple times.

## Architecture Benefits

✅ **Single Source of Truth**
- Plan creation logic lives in orchestrator
- Both HTTP and WebSocket use same code path

✅ **No Code Duplication**
- Business logic not repeated per UI
- Console UI, Web UI, future APIs all use orchestrator

✅ **Testable**
- Can test orchestrator methods directly
- HTTP endpoints become trivial wrappers

✅ **Consistent**
- Same validation, error handling, state transitions everywhere
- Reduces bugs from implementation differences

✅ **Maintainable**
- Changes to business logic happen in one place
- Clear separation: HTTP layer = translation, orchestrator = logic
