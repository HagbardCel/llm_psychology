# Assessment Stability and LLM Caching Implementation Plan (Reworked)

**Date:** January 6, 2026
**Status:** Planned
**Related Incident:** Missing Assessment Recommendations (Quota Exhaustion)

## 1. Executive Summary
This revision aligns the plan with project design principles (structured outputs, DI, and fail-fast error handling) and removes unnecessary complexity. The main changes are:
1. Replace per-style parallel calls with a **single structured-output assessment call** to cut request volume to 1 and avoid JSON scraping.
2. Add an **optional LLM response cache** with safe keying, retention controls, and DI-based wiring.
3. Handle assessment job failures with **quota-specific catch-and-continue** (allowed by design rules) and **fail-fast re-raise** for all other errors.

## 2. Updated Root Cause Analysis
- Assessment recommendations are generated by spawning multiple LLM calls in parallel (one per style). This multiplies quota usage and triggers `LLMQuotaExhaustedError` quickly.
- `_run_assessment_job` runs inside a Trio nursery without error handling. Any exception cancels sibling tasks and can disconnect the WebSocket before a user-visible `error` message is sent.
- Sequentializing the existing per-style calls would reduce concurrency but **not** reduce total quota usage; it still makes N requests and does not align with the preference for structured outputs.

## 3. Goals and Non-Goals
**Goals**
- Reduce assessment LLM calls from N to 1 while keeping recommendation quality.
- Add optional response caching without violating Trio or DI patterns.
- Ensure quota exhaustion results in a user-visible error without disconnecting the session.
- Keep external HTTP/WS contracts unchanged.

**Non-Goals**
- No new WS message types or payload fields.
- No change to therapy style selection flow or DTOs.

## 4. Implementation Plan

### Phase 1: Single Structured-Output Assessment Call (Required)
**Goal:** Eliminate per-style parallel calls and use schema-guided output, per design rules.

**Design**
- Add a Pydantic model in `src/psychoanalyst_app/models/structured_output_models.py`:
  - `AssessmentRecommendationsOutput` with `recommendations: list[AssessmentRecommendation]`
  - Each entry includes `style_id`, `score`, `assessment`, `key_topics`.
- Update `TrioAssessmentAgent._generate_recommendations` to:
  - Build a single prompt containing the session summary and a list of available styles (id + assessment prompt + description).
  - Call `LLMService.generate_structured_output_async(..., schema=AssessmentRecommendationsOutput)`.
  - Validate style ids and normalize scores (0.0-1.0) in code (fail-fast on schema mismatch).
  - Remove the per-style `trio.open_nursery()` and `_assess_style` path.

**Notes**
- This aligns with `docs/design-principles.md` (prefer structured outputs, avoid JSON scraping).
- The assessment output remains internal; no schema generation or frontend type updates required.

### Phase 2: LLM Response Cache (Required)
**Goal:** Allow cost-free retries for identical prompts without violating Trio design rules.

**Design**
- **Persistence:** Implement `LLMCacheRepo` in `src/psychoanalyst_app/services/db/repos/` to manage the `llm_cache` table. Access it via `TrioDatabaseService` to maintain the "one persistence abstraction" rule.
- **Migration:** Add a migration in `src/psychoanalyst_app/services/migration_service.py` to create the `llm_cache` table.
  - Update `_migration_001_initial_schema` for clean DB resets.
  - Add a new migration function (next version) and register it in `_get_migrations()`.
  - Schema: `cache_key TEXT PRIMARY KEY`, `call_type TEXT`, `model_name TEXT`, `prompt TEXT`, `context_json TEXT`, `schema_hash TEXT`, `response_json TEXT`, `created_at TEXT`, `user_id TEXT`, `session_id TEXT`, `source TEXT`.
- **Service:** Add `LLMCacheService` (`src/psychoanalyst_app/services/llm_cache_service.py`) as a logic layer over the repo (hashing, policy checks).
- **Integration:** Update `LLMService` **async wrappers only** (`generate_response_async`, `generate_structured_output_async`):
  - Check cache before rate limiting.
  - Store response after successful LLM call.
  - Cache operations are fail-fast when caching is enabled (log then re-raise).
- **Keys:** Cache key must include:
  - `call_type` (response vs structured output), `model_name`, `prompt`, `context`,
  - `schema_hash` and `method` for structured outputs,
  - `LLM_CALL_CONTEXT` identifiers (user_id/session_id/source) to avoid cross-user reuse.
  - If required context identifiers are missing or source is not allowlisted, skip caching.
- **Storage:** Keep full prompt/context/response payloads for now to ease debugging; revisit if retention or privacy constraints change.

**Configuration**
- Add settings in `src/psychoanalyst_app/config.py`:
  - `LLM_CACHE_ENABLED` (default False in production, True in dev/usertest)
  - `LLM_CACHE_MAX_AGE_DAYS` (TTL purge)
  - `LLM_CACHE_MAX_ROWS` (size bound)
  - Optional `LLM_CACHE_SOURCES` allowlist (default: "assessment")
  - Optional `LLM_CACHE_REQUIRE_CONTEXT` (default True) to enforce user/session/source presence.
  - Wire `LLMCacheService` into the container and pass it into `LLMService` via DI (no module-level singletons).

### Phase 3: Assessment Job Error Handling (Required)
**Goal:** Surface errors to the user without violating fail-fast rules.

**Design**
- Update `_run_assessment_job` in `src/psychoanalyst_app/orchestration/orchestrator_helpers.py`:
  - Catch `LLMQuotaExhaustedError` explicitly:
    - Prefer the existing quota-exhausted event emission from `LLMService` (handled in `src/psychoanalyst_app/trio_server.py`).
    - If emitting here, guard with a per-session flag to avoid duplicate `error` messages.
    - Return without re-raising (permitted exception to keep workflow alive).
  - For all other exceptions:
    - Send a best-effort `error` message.
    - Re-raise to preserve fail-fast behavior.
- Keep workflow state in `ASSESSMENT_IN_PROGRESS` on quota exhaustion so the job can retry on reconnect or next `ensure_assessment_job` call.

## 5. Verification Plan
1. **Single-call assessment**
   - Trigger assessment and confirm only one LLM request for recommendations.
   - Validate output is schema-compliant and ordered/normalized.
2. **Cache behavior**
   - Run an assessment, confirm cache entry is written (Docker: `docker compose run --rm api sqlite3 data/psychoanalyst.db "select count(*) from llm_cache;"`).
   - Repeat assessment with identical input and verify it is served from cache (check logs for "Cache HIT").
3. **Quota error handling**
   - Force `LLMQuotaExhaustedError` and confirm:
     - `error` message is sent to client.
     - WebSocket stays connected.
     - State remains in `ASSESSMENT_IN_PROGRESS`.
4. **Non-quota error handling**
   - Simulate a non-quota exception in assessment:
     - `error` message is attempted.
     - Exception re-raises (fail-fast) and crashes the nursery as expected.

## 6. Alignment with Design Principles
- **Structured outputs first:** Single structured-output assessment replaces JSON scraping.
- **Trio-first + DI:** Cache operations use the existing DB executor pattern; no new raw SQL connections in the event loop.
- **Fail-fast behavior:** Only quota exhaustion is caught and continued; other errors re-raise after user notification.
- **Stable contracts:** Reuse existing `error` messages; no new WS/HTTP schemas.

## 7. Execution Steps
1. Define `AssessmentRecommendationsOutput` and update prompt logic in `TrioAssessmentAgent`.
2. Create `llm_cache` migration, `LLMCacheRepo`, and `LLMCacheService`.
3. Wire caching into `LLMService` (async methods only).
4. Update `orchestrator_helpers.py` with specific error handling logic.
